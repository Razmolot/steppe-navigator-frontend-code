# ТЗ: Массовая генерация профориентационных отчётов по школе (ИИ-сферы + отчёт) с прогрессом

**Проект:** Steppe Navigator / Платформа профориентатора

**Цель:** дать профориентатору возможность в 1–2 действия запустить генерацию отчётов для *всех* учеников школы, которые готовы (все 4 диагностики завершены) и у которых ещё нет отчёта, с прозрачным прогрессом и контролируемой нагрузкой.

---

## TL;DR (что меняем)

1) Добавляем сущность **Bulk Job** (задание массовой генерации) + прогресс (progress bar, счётчики, ошибки по ученикам).
2) Генерация для каждого ученика выполняется пайплайном:
   - Проверка готовности (4 теста completed) и отсутствия отчёта → иначе `skipped`
   - `ai-spheres` (получить 3 сферы)
   - `save spheres` (создать/обновить **draft** в `career_reports.selected_spheres`, `status=draft`, `spheres_from_ai=true`)
   - `generate report` (подбор профессий + статус `generated`)
3) **ИИ‑резюме НЕ генерируется в bulk** (только в ручном сценарии).

---

## Контекст и текущий флоу (как сейчас)

Текущий путь (индивидуальная работа) требует много кликов:
- кнопка «Профориентация» → выбор сфер → (опционально ИИ) → «Сохранить сферы» → «Сгенерировать отчёт».

Сферы уже хранятся до генерации:
- `career_reports.selected_spheres` (json), статус `draft`.
- API сохранения: `POST /api/counselor/career/students/{studentId}/spheres`.

Отчёт генерируется из сохранённых сфер:
- API: `POST /api/counselor/career/reports/{reportId}/generate`.

Проблема регенерации: при «Сгенерировать заново» пользователь попадает на «чистый лист», хотя технически сферы можно и нужно предзаполнять из draft/последнего отчёта.

## Требования

### Роли и доступ
- Запускать bulk может **только профориентатор** (роль `career_counselor`).
- Ограничение по доступу: профориентатор должен быть привязан к школе (через текущую механику `canAccessStudent`/`schools()`), и bulk работает **в пределах одной школы**.

### Скоуп обработки
Bulk запускается **по всей школе** и обрабатывает только учеников:
- `role='student'` и `is_active=1`
- есть completed-сессии по всем 4 тестам: `riasec`, `soft_skills`, `high5`, `questionnaire`
- **нет существующего отчёта** (см. раздел «Правила пропуска»)

### Правила пропуска (строго)
- Если у ученика уже есть отчёт (status `generated|completed`) → **пропустить**, ничего не перетирать.
- Если у ученика нет готовности (не завершены 4 теста) → пропустить.
- Если во время обработки отчёт появился (гонка) → пропустить.

### Сохранение сфер (обязательное)
- Сферы, полученные от ИИ, **обязательно сохраняются** до генерации отчёта (как draft).
- Цель: чтобы при дальнейшем действии «выбрать другие сферы» (вместо «сгенерировать заново») UI открывался **не с пустого листа**, а с текущими 3 сферами.

### ИИ‑резюме
- В bulk-режиме **ИИ‑резюме не генерируем**.
- В ручном режиме поведение остаётся как есть (или отдельно настраивается — см. «Риски/купирование»).

## UX / UI

### A1) Bulk‑генерация (по школе)

#### Где расположить кнопку
На уровне школы (страница школы/учеников школы):
- Кнопка: **«Сгенерировать отчёты (ИИ) для школы»**

#### Диалог подтверждения
Перед запуском показать:
- Кол-во найденных кандидатов (готов + без отчёта)
- Пояснение: «Пропустим учеников, у которых уже есть отчёт»
- Кнопки: `Запустить` / `Отмена`

#### Экран прогресса
После запуска делаем редирект на страницу прогресса:
- `/app/counselor/career/bulk-jobs/{jobId}`

Содержимое страницы:
- Progress bar: `processed/total`
- Счётчики: `успешно`, `пропущено`, `ошибка`
- Таблица последних событий/ошибок по ученикам (минимум: имя/класс/статус/ошибка + ссылка «Открыть ученика»)
- Автообновление (polling) раз в 2–3 секунды, плюс кнопка `Обновить` (fallback)

#### Поведение при закрытии страницы
- Job продолжает выполняться.
- При повторном заходе профориентатор может открыть job и увидеть статус/прогресс.

### B1) ИИ‑резюме (только ручной режим)

На странице отчёта добавляем блок «ИИ‑резюме» и кнопку **«Сгенерировать ИИ‑резюме»**.

Состояния UI:
- Резюме отсутствует → кнопка «Сгенерировать ИИ‑резюме»
- Генерация идёт → disabled‑кнопка + статус «Генерируем…»
- Резюме готово → показываем текст (ru/kk/en), кнопка «Сгенерировать заново» (перезаписывает текст)
- Ошибка → текст ошибки + кнопка «Повторить»

Примечание: bulk‑генерация этот endpoint не вызывает.

### Действие в отчёте: смена сфер
На странице отчёта добавляем действие **«Выбрать другие сферы»**.
Оно открывает экран выбора сфер, **предзаполненный** текущими `selected_spheres`.

## API (предложение)

### 1) Запуск bulk
`POST /api/counselor/career/schools/{schoolId}/bulk-generate`

Тело (минимально):
```json
{}
```

Ответ:
```json
{
  "job_id": "uuid",
  "total": 123
}
```

### 2) Прогресс bulk
`GET /api/counselor/career/bulk-jobs/{jobId}`

Ответ:
```json
{
  "job": {
    "id": "uuid",
    "status": "queued|running|completed|failed|canceled",
    "total": 123,
    "processed": 10,
    "success": 7,
    "skipped": 2,
    "failed": 1,
    "started_at": "...",
    "finished_at": null
  },
  "items": [
    {
      "student_id": 3,
      "status": "done|skipped|failed",
      "error": null
    }
  ]
}
```

### 3) (Опционально) отмена
`POST /api/counselor/career/bulk-jobs/{jobId}/cancel`
- помечает job как `canceled_requested`, worker уважает флаг и корректно завершает.

### `career_bulk_jobs`
- `id` (uuid)
- `school_id`
- `counselor_id`
- `status` enum: `queued|running|completed|failed|canceled`
- `total` int
- `processed` int
- `success` int
- `skipped` int

Индексы:
- (`school_id`, `created_at`)
- (`counselor_id`, `created_at`)

- `id` (bigint) или uuid
- `job_id` uuid (FK)

- `error` text/json
- timestamps

Индексы:
- (`job_id`, `status`)
- unique (`job_id`, `student_id`)

## Очереди и выполнение

### Очередь
- Вводим отдельную очередь, например `career_ai`.
- Запускаем фиксированное число воркеров (например 1–3) на весь сервер/кластер.

### Структура jobs
Мини‑max вариант (простой и безопасный):
- 1 Laravel Job «ProcessCareerBulkJob»
  - загружает список кандидатов
  - создаёт items со статусом `queued`
  - по одному (или маленькими пачками) обрабатывает items

Альтернатива (масштабнее):
- Job‑оркестратор создаёт отдельные `ProcessCareerBulkStudentItem` jobs на каждого студента.

### Алгоритм обработки одного ученика
1) Повторно проверить eligibility:
   - 4 completed
   - отчёта `generated|completed` нет
2) Запросить сферы от ИИ (SpheresService / текущий endpoint)
3) Сохранить сферы:
   - вызвать текущий `createOrUpdateDraft(studentId, counselorId, spheres, fromAi=true)`
4) Сгенерировать отчёт:
   - вызвать `generateReport(reportId)`
5) Обновить item status + инкрементировать счётчики

### Retry/backoff
- Для ошибок OpenAI/HTTP:
  - `429` → экспоненциальный backoff + ограничение параллелизма
  - `5xx` → retry ограниченное число раз
  - в случае исчерпания попыток → item `failed`, продолжаем дальше

## Важные ограничения и зоны риска (и как купировать)

### Риск 1: лимиты/стоимость OpenAI при массовых запусках
**Симптом:** 429/timeout, очередь разрастается, пользователи нервничают.

**Купирование:**
- отдельная очередь `career_ai`
- лимит воркеров (1–3)
- backoff/retry
- прогресс-бар и честные статусы
- НЕ генерировать AI‑резюме в bulk

### Риск 2: гонки и идемпотентность (две кнопки нажали, два профориентатора, параллельные job’ы)
**Симптом:** дубль‑отчётов, перетирание сфер.

**Купирование:**
- повторная проверка «отчёта нет» перед обработкой item
- атомарная блокировка на student_id (опционально: DB lock / cache lock)
- unique ограничение на draft per student (сейчас логика: один draft на student)

### Риск 3: регресс UX “пустой лист при пересборке/смене сфер”
**Симптом:** потеря сфер, лишние клики.

**Купирование (обязательные критерии):**
- bulk всегда сохраняет spheres в draft
- действие «выбрать другие сферы» открывает UI с предзаполнением из `career_reports.selected_spheres` (draft или latest)
- “пересобрать отчёт” не должен очищать selected_spheres

### Риск 4: долгая операция без обратной связи
**Симптом:** люди нажимают повторно, думают что «зависло».

**Купирование:**
- Bulk Job хранит прогресс
- UI polling и явные статусы
- список ошибок/пропусков

### Риск 5: рост БД и логов
**Купирование:**
- ограничить хранение job’ов (например, авто‑архив/удаление старше 30–90 дней)
- ограничить хранение items (или хранить только failed/skipped + агрегаты)

## Acceptance Criteria

1) Профориентатор может запустить bulk по школе.
2) Bulk обрабатывает только учеников: 4 теста completed + нет generated/completed отчёта.
3) Для каждого обработанного ученика:
   - создаются/обновляются сферы в draft `career_reports.selected_spheres`, `status=draft`, `spheres_from_ai=true`
   - генерируется отчёт (`status=generated`)
4) Bulk пропускает учеников с уже существующим отчётом и не перетирает данные.
5) UI: после запуска происходит редирект на страницу прогресса job’а, где видны progress bar + счётчики + ошибки.
6) В bulk **не генерируется AI‑резюме**.
7) UI отчёта: есть действие **«Выбрать другие сферы»**, которое открывает выбор сфер *предзаполненный* текущими `selected_spheres`.
8) UI отчёта: есть ручная кнопка **«Сгенерировать ИИ‑резюме»**, которая вызывает отдельный endpoint `.../ai-resume` и показывает состояния (нет/генерируется/готово/ошибка).

## Рекомендуемая поэтапная реализация (опционально)

### Этап 0: dry-run
- кнопка + подсчёт кандидатов + создание job без обработки

### Этап 1: job + прогресс + обработка без OpenAI
- вместо AI spheres использовать заглушку (например, фиксированный набор сфер) → проверить стабильность очереди/прогресса

### Этап 2: подключить AI spheres
- включить вызов `ai-spheres`
- настроить retries/backoff

### Этап 3: подключить generate report
- финальный пайплайн ai-spheres → save draft → generate

### Этап 4: UX смены сфер (без «пересобрать отчёт»)
- В отчёте заменить текущее действие «Сгенерировать заново» на **«Выбрать другие сферы»**.
- «Выбрать другие сферы» ведёт на экран выбора сфер, **предзаполненный** сохранёнными сферами (`career_reports.selected_spheres`).
- Отдельной кнопки «Пересобрать отчёт» **не добавляем**.

## Примечания по существующему коду

- Draft сферы уже поддерживаются через:
  - `CareerReportService::createOrUpdateDraft()`
  - `POST /api/counselor/career/students/{studentId}/spheres`
- Генерация отчёта использует `selected_spheres`:
  - `CareerReportService::generateReport(reportId)`

**Важно (принятое решение):** текущий `CareerGuidanceController::generateReport()` после генерации вызывает `generateAiResume()`.

Чтобы bulk не генерировал резюме (и не удваивал нагрузку на OpenAI), **принято решение №2: вынести генерацию AI‑резюме в отдельный endpoint**, а `generateReport` делать «только профессии/вектор/статус».

Требуемые изменения:
- `POST /api/counselor/career/reports/{reportId}/generate`:
  - больше **не** вызывает `generateAiResume()`.
- Новый endpoint (ручной режим):
  - `POST /api/counselor/career/reports/{reportId}/ai-resume` → запускает `generateAiResume(reportId)`
  - доступ: только владелец отчёта (counselor) и только для отчётов `generated|completed`
  - UI вызывает этот endpoint только в ручном сценарии (кнопка «Сгенерировать ИИ‑резюме»).

Риск/купирование:
- изменение затрагивает ручной UX: нужно добавить кнопку/действие для резюме;
- bulk становится стабильнее и дешевле.
